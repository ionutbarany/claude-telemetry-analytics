# Data Model — Claude Code Telemetry Analytics

Star-schema design for storing and querying Claude Code OpenTelemetry events in PostgreSQL. Tables are sized for direct translation into SQLAlchemy 2.0 mapped classes and Alembic migrations.

Source event families (Claude Code OTel logs):

| Source event | Target table |
|--------------|--------------|
| `claude_code.user_prompt` | `fact_user_prompts` |
| `claude_code.api_request` | `fact_api_requests` |
| `claude_code.tool_result` | `fact_tool_results` |
| `claude_code.api_error` | `fact_api_errors` |

Shared OTel attributes (`session.id`, `user.*`, `organization.id`, `app.version`, …) land in dimensions; measures and event-specific attributes land in facts.

**Grain notes**

- Facts are additive at *one row per source event*.
- `prompt_id` (UUID) correlates rows across fact tables for a single user turn; it is a degenerate dimension (stored on facts, not its own table).
- `dim_sessions` is a slowly changing / append-style dimension keyed by Claude Code `session.id`.
- `dim_models` is a small reference dimension keyed by model identifier string.

## Conventions

| Convention | Rule |
|------------|------|
| Surrogate keys | `*_sk` — `BIGINT GENERATED BY DEFAULT AS IDENTITY` (or sequence) |
| Natural keys | Source IDs preserved as `*_nk` or domain names (`session_id`, `account_uuid`) |
| Timestamps | `TIMESTAMPTZ` (UTC); event time = `event_ts`, load time = `ingested_at` |
| Money | `NUMERIC(12, 6)` for USD costs |
| Booleans | PostgreSQL `BOOLEAN` (ETL maps OTel `"true"`/`"false"` strings) |
| JSON payloads | `JSONB` for optional / sparse structured fields |
| Soft deletes | Not used on facts; dimensions may set `is_active` |
| Naming | `snake_case` tables and columns |

All fact tables include:

- `user_sk`, `session_sk` foreign keys
- `event_ts`, `event_sequence`, `prompt_id` for ordering and turn correlation
- `ingested_at` for ETL audit
- `source_event_name` for lineage back to the OTel event type

## Dimension tables

### `dim_users`

One row per distinct Claude Code user identity. Prefer `account_uuid` when authenticated; fall back to anonymous `user_id` for unauthenticated installs.

Telemetry events supply identity fields (`user.id`, `user.account_uuid`, email, org). Organizational attributes — `practice`, `level`, and `location` — come from `employees.csv` and are joined onto the user dimension during ETL (typically by email or account id). Those columns enable organizational analytics such as usage and cost by engineering practice, seniority band, or country, without bloating fact tables.

| Column | PostgreSQL type | Nullable | Description |
|--------|-----------------|----------|-------------|
| `user_sk` | `BIGINT GENERATED BY DEFAULT AS IDENTITY` | NO | Surrogate primary key |
| `user_id` | `TEXT` | NO | Claude Code anonymous install ID (`user.id`), or IdP subject on gateway sessions |
| `account_uuid` | `UUID` | YES | Authenticated account UUID (`user.account_uuid`) |
| `account_id` | `TEXT` | YES | Anthropic admin-style account id (`user.account_id`, e.g. `user_01…`) |
| `email` | `CITEXT` | YES | OAuth / gateway email (`user.email`); use `CITEXT` or `TEXT`; join key to `employees.csv` |
| `organization_id` | `UUID` | YES | Organization UUID (`organization.id`) |
| `identity_source` | `TEXT` | YES | e.g. `gateway-oidc` when stamped by apps gateway |
| `user_groups` | `TEXT` | YES | Comma-separated IdP groups when present |
| `practice` | `TEXT` | YES | Engineering practice from `employees.csv` (e.g. Platform, Data, ML, Backend, Frontend) |
| `level` | `TEXT` | YES | Engineering seniority from `employees.csv` (L1–L10) |
| `location` | `TEXT` | YES | Employee country/location from `employees.csv` |
| `first_seen_at` | `TIMESTAMPTZ` | NO | Earliest observed event timestamp for this user |
| `last_seen_at` | `TIMESTAMPTZ` | NO | Most recent observed event timestamp |
| `is_active` | `BOOLEAN` | NO | Soft flag for analytics filters; default `TRUE` |
| `created_at` | `TIMESTAMPTZ` | NO | Row insert time; default `NOW()` |
| `updated_at` | `TIMESTAMPTZ` | NO | Last dimension update; default `NOW()` |

**Primary key:** `user_sk`

**Foreign keys:** none

**Natural key / uniqueness**

- Unique index on `account_uuid` where not null.
- Unique index on `user_id` (always present on source events).
- ETL upsert preference: match `account_uuid` first, else `user_id`.
- Enrich from `employees.csv` after identity upsert; leave `practice` / `level` / `location` null when no employee match.

**SQLAlchemy notes**

- Map `user_sk` as `mapped_column(BigInteger, Identity(), primary_key=True)`.
- Use `UniqueConstraint` / partial unique indexes in Alembic for `account_uuid`.
- Optional indexes on `(practice)`, `(level)`, and `(location)` for organizational rollups.

## Cross-fact correlation

Typical turn reconstruction:

```sql
SELECT p.prompt_id, p.event_ts AS prompt_ts,
       r.request_id, r.cost_usd, r.input_tokens, r.output_tokens,
       t.tool_name, t.success, t.duration_ms,
       e.status_code, e.error_message
FROM fact_user_prompts p
LEFT JOIN fact_api_requests r
  ON r.prompt_id = p.prompt_id AND r.session_sk = p.session_sk
LEFT JOIN fact_tool_results t
  ON t.prompt_id = p.prompt_id AND t.session_sk = p.session_sk
LEFT JOIN fact_api_errors e
  ON e.prompt_id = p.prompt_id AND e.session_sk = p.session_sk
WHERE p.session_sk = :session_sk
ORDER BY p.event_sequence, r.event_sequence, t.event_sequence;
```

`prompt_id` is intentionally a degenerate dimension: cardinality is unbounded and it only exists on events, not metrics.

## Why a star schema

1. **Query shape matches the product.** Dashboards and APIs ask dimensional questions: cost by model, errors by user, tool latency by session, prompts per day. A star lets those filters hit thin dimensions and scan only the relevant fact.
2. **Additive measures stay clean.** Tokens, cost, durations, and success flags live on facts with a clear grain. Dimensions hold descriptive attributes that change slowly (email, app version, model family).
3. **Join cost is predictable.** Facts join dimensions on integer surrogate keys; BI and SQLAlchemy queries stay simple (`JOIN dim_models USING (model_sk)`).
4. **ETL is explicit.** Polars transforms map each OTel event type to one fact loader and shared dimension upserts — easier to test and idempotent than mutating a polymorphic dump.
5. **Interview / production narrative.** Star schemas are the standard for analytics warehouses; this design shows intentional modeling rather than “store JSON and hope.”

---

## Why a single wide events table was rejected

A single `events` table with `event_type` + sparse columns (or one giant `JSONB` payload) was considered and rejected for v1 curated analytics:

| Problem | Impact |
|---------|--------|
| **Sparse columns** | `cost_usd` / `tool_name` / `status_code` mostly null depending on type → wasted storage, weak typing, awkward CHECK constraints |
| **Heterogeneous grain** | Mixing prompts, API calls, tools, and errors forces every query to filter `event_type` and re-document grain |
| **Weak constraints** | Cannot enforce “API requests must have `model_sk`” or “tool results must have `tool_name`” without complex partial constraints |
| **Index bloat** | One table needs indexes for every access path; unused indexes still pay write cost for every event type |
| **Partition prune noise** | Time partitions help, but type-filtered scans still touch pages for irrelevant event kinds |
| **Analytics friction** | Cost rollups accidentally include error rows; tool latency queries must remember which columns apply |

Staging may still keep a raw/near-source event log for reprocessing. The **curated** layer uses typed fact tables so FastAPI services and Streamlit charts query intentional grains.

## Indexing strategy

### Dimensions

| Table | Index | Purpose |
|-------|-------|---------|
| `dim_users` | `UNIQUE (user_id)` | Idempotent upsert by install id |
| `dim_users` | `UNIQUE (account_uuid) WHERE account_uuid IS NOT NULL` | Prefer authenticated identity |
| `dim_users` | `(organization_id)` | Org-scoped user lists |
| `dim_users` | `(email)` where used for lookup | Optional admin search / `employees.csv` join |
| `dim_users` | `(practice)`, `(level)`, `(location)` | Organizational analytics rollups |
| `dim_sessions` | `UNIQUE (session_id)` | Natural key upsert |
| `dim_sessions` | `(user_sk, started_at DESC)` | Sessions for a user over time |
| `dim_models` | `UNIQUE (model_id)` | Natural key upsert |
| `dim_models` | `(model_family)` | Family rollups |

### Facts (all fact tables)

Shared patterns (substitute table/PK name):

| Index | Columns | Purpose |
|-------|---------|---------|
| Time | `(event_ts DESC)` | Dashboard time windows; partition-local |
| Session timeline | `(session_sk, event_sequence)` | Reconstruct a session in order |
| User + time | `(user_sk, event_ts DESC)` | Per-user activity |
| Turn correlation | `(prompt_id) WHERE prompt_id IS NOT NULL` | Join facts for one prompt |
| Ingest audit | `(ingested_at)` | ETL lag / reprocess windows |

### Fact-specific

| Table | Index | Purpose |
|-------|-------|---------|
| `fact_api_requests` | `(model_sk, event_ts DESC)` | Cost/tokens by model over time |
| `fact_api_requests` | `(request_id) WHERE request_id IS NOT NULL` | Join to Anthropic support / traces |
| `fact_api_requests` | `(query_source, event_ts DESC)` | Main vs subagent vs auxiliary |
| `fact_tool_results` | `(tool_name, event_ts DESC)` | Tool popularity and latency |
| `fact_tool_results` | `(success, event_ts DESC)` | Failure rates |
| `fact_tool_results` | `(tool_use_id) WHERE tool_use_id IS NOT NULL` | Hook / decision correlation |
| `fact_user_prompts` | `(session_sk, prompt_id) UNIQUE WHERE prompt_id IS NOT NULL` | Idempotent prompt loads |
| `fact_user_prompts` | `(command_name) WHERE command_name IS NOT NULL` | Command usage |
| `fact_api_errors` | `(status_code, event_ts DESC)` | HTTP error dashboards |
| `fact_api_errors` | `(model_sk, event_ts DESC)` | Error rate by model |

**Guidelines**

- Prefer **partial indexes** for sparse nullable keys (`prompt_id`, `request_id`).
- Avoid indexing high-churn `TEXT` bodies (`prompt_text`, `error_message`).
- For `JSONB` (`tool_parameters`), add expression/GIN indexes only after a concrete query need appears.
- Keep write-path indexes minimal during bulk ETL; create non-essential indexes `CONCURRENTLY` after backfill if needed.

## Partitioning strategy

Telemetry facts grow without bound. Partition **fact tables only** by time; keep dimensions unpartitioned.

### Approach

- **Method:** PostgreSQL declarative range partitioning on `event_ts`.
- **Interval:** monthly partitions for moderate volume; weekly if a single month exceeds ~50–100M rows or vacuum/index maintenance becomes painful.
- **Naming:** `fact_api_requests_YYYY_MM` (e.g. `fact_api_requests_2026_07`).
- **Primary key:** `(api_request_sk, event_ts)` (and analogues) so the partition key is part of the PK.
- **Foreign keys from facts → dimensions:** supported; FKs *to* partitioned facts are avoided (dimensions do not reference facts).

Example parent definition sketch:

```sql
CREATE TABLE fact_api_requests (
    api_request_sk  BIGINT GENERATED BY DEFAULT AS IDENTITY,
    event_ts        TIMESTAMPTZ NOT NULL,
    -- … remaining columns …
    PRIMARY KEY (api_request_sk, event_ts),
    FOREIGN KEY (user_sk) REFERENCES dim_users (user_sk),
    FOREIGN KEY (session_sk) REFERENCES dim_sessions (session_sk),
    FOREIGN KEY (model_sk) REFERENCES dim_models (model_sk)
) PARTITION BY RANGE (event_ts);

CREATE TABLE fact_api_requests_2026_07
    PARTITION OF fact_api_requests
    FOR VALUES FROM ('2026-07-01') TO ('2026-08-01');
```

Apply the same pattern to `fact_tool_results`, `fact_user_prompts`, and `fact_api_errors`.

### Retention and maintenance

| Operation | Practice |
|-----------|----------|
| **Drop old data** | `DETACH PARTITION` + `DROP` (or archive to object storage) instead of large `DELETE`s |
| **Create ahead** | Scheduler/migration creates next 1–2 months of partitions |
| **Default partition** | Optional `DEFAULT` partition to catch clock-skew outliers; alert and move rows |
| **Indexes** | Create indexes on the parent so they propagate; verify partition-local indexes after major PG upgrades |
| **Analytics** | Always include `event_ts` predicates so partition pruning applies |

### When to introduce partitioning

Ship unpartitioned tables for early demos if volume is small, but **define models with `event_ts` in the PK-ready shape** so Alembic can convert to partitioned parents without rewriting application queries. Convert once daily inserts sustainably exceed a few million rows or query latency on time filters degrades.

## SQLAlchemy implementation checklist

Use these mappings when creating `app/db/models` (names illustrative):

| DB table | Mapped class | Notes |
|----------|--------------|-------|
| `dim_users` | `DimUser` | `Identity()` PK; unique on `user_id` |
| `dim_sessions` | `DimSession` | FK `user_sk`; unique `session_id` |
| `dim_models` | `DimModel` | unique `model_id` |
| `fact_api_requests` | `FactApiRequest` | FKs to user, session, model |
| `fact_tool_results` | `FactToolResult` | FKs to user, session; `Mapped[dict \| None]` for JSONB |
| `fact_user_prompts` | `FactUserPrompt` | FKs to user, session |
| `fact_api_errors` | `FactApiError` | FKs to user, session; optional model |

Shared patterns:

- `MappedAsDataclass` optional; prefer explicit `mapped_column` + docstrings on relationships.
- `DateTime(timezone=True)` for all timestamps.
- `Numeric(12, 6)` for `cost_usd`.
- `sqlalchemy.Dialect.kwargs` / Alembic: `postgresql_partition_by='RANGE (event_ts)'` when enabling partitions.
- Session factory and repositories live in services; models stay persistence-only.

Suggested relationship sketch:

```text
DimUser.sessions          → list[DimSession]
DimSession.user           → DimUser
DimSession.api_requests   → list[FactApiRequest]
DimSession.tool_results   → list[FactToolResult]
DimSession.user_prompts   → list[FactUserPrompt]
DimSession.api_errors     → list[FactApiError]
DimModel.api_requests     → list[FactApiRequest]
DimModel.api_errors       → list[FactApiError]
```

Do not define ORM relationships between fact tables solely on `prompt_id` (no FK). Correlate in SQL/services with explicit joins.
